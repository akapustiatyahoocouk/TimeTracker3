Activities and Tasks

TimeTracker3 uses Activities and Tasks to record the kinds of activity users can spend their working time on.

Activities

An Activity is an indication of what kind of work the user can be performing at some given moment. Examples of Activities are such as "Meetings", "Writing documentation", or even "Lunch break". Once a TimeTracker workspace is properly set up, the user can indicate the moment when they started or stopped the corresponding Activity by a single mouse click (or, if the user's screen has touch recognition functions, a single tap on the screen). In this the TimeTracker is very similar to the classical chess clock - only the number of buttons on the clock can vary with the time, based on what any specific user does during  different periods of time.

For an Activity, the following information is kept:

  • Name - a short descriptive name identifying the specific Activity.
  • Description - a longer description, providing additional information on what a specific Activity is all about. For some Activities (such as "Lunch break") the description will not even be necessary while for others (such as "Code refactoring" and "Performance improvements") the description can e.g. provide guidelines as to what actions are considered belonging to each of the Activities.
  • Activity type - all Activities can, and usually will, be associated with specific Activity Types (see below). These provide categorisation of Activities into broad classes (such as "Development", "Documentation", "Meetings", "R&R", etc.)
  • Workload - an Activity can be associated with a "workload" (see below), which represents a work stream towards which time spent on that Activity contributes. This allows collecting information on how much overall time was dedicated to each project, thus helping to better manage the working time of individuals and teams.
  • Commenting requirements - an Activity may be configured to require the user to enter a comment when starting and/or stopping the activity. Examples of activity start/stop comments are situations like "Record the reason for starting code refactoring" or "Record the caller's name after an urgent support-related phone call".
  • Reminder requirements - an Activity may be configured to present a full-screen reminder while that Activity is underway. A typical example here would be a "Lunch break" or "Meeting" Activities, where the user goes away from their computer and must, when returning to it, not forget to indicate that the away time is now over and the actual work begins again.
  • Timeout - when the user leaves for good (for example, at the end of a working day), they may sometimes forget to "stop" their current activity, which will then continue ticking overnight, thus introducing inaccuracy to working time measurement. To prevent that, an Activity may be configured with an "inactivity timeout" - a period of time after which, if there is no user activity on the computer where TimeTracker3 runs, the latter considers the user as having left and automatically records the end of the currently ticking Activity at the date and time when the last user activity was noticed. NOTE THAT for Activities not explicitly configured with an "inactivity timeout" the default timeout can also be configured on a per-User bases.

Tasks

Tasks are very similar to Activities in that they also indicate what the user's efforts are being spent on. Tasks are different from Activities in two respects:

  • Unlike Activities, which are organised into a simple list from which the "current" Activity can be chosen to work on, Tasks are organised into a hierarchy, where upper-level tasks can have sub-tasks which, in turn, can have their own sub-tasks, and so on.
  • A Task has, in addition to all the properties of an Activity, a completion status, which is an indication of whether that Task has been completed or not. This simplifies user's lives by separating the Tasks-to-be-completed from the Tasks-already-completed; TimeTracker can (and usually will) be configured to show only the not-yet-completed Tasks so that the user can better dedicate the working time to those.
  • A Task also has a commenting requirement - when a user marks a given Task as "completed", the Task in question may require the user to enter the short comment, typically about why the Task is being marked as "completed". Typical comments will be e.g. "done and tested", "no longer relevant" or even "Task rejected after a meeting".

Public and private Activities and Tasks

Both Activities and Tasks can be "public" or "private":

  • A public Activity or Task is visible to all users who can login to the TimeTracker3 workspace. These are organisation-wide things like a "Lunch break" Activity or "Preparation for a 2025 world conference on XYZ" Task.
  • A private Activity or Task are visible only to the User to whom they belong. These represent the personal backlog of that User.
